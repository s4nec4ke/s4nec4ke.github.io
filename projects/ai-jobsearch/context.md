# ai_jobsearch

## Одна строка (для превью на главной)
Автоматизировал весь job-search с нуля: Telegram-парсер вакансий → AI-скоринг → подбор резюме → готовые письма HR → трекинг ответов → подготовка к интервью.

## Что это и зачем
Pet-project-инструмент для собственного поиска работы. Боль: вручную мониторить десятки Telegram-каналов с вакансиями, оценивать каждую под свой профиль, адаптировать резюме, писать персонализированные письма — это часы в день с низкой ценностью.

Решение: pipeline из Python-парсеров + Claude Code-агентов. Каждый день одна команда (`@hr-daily`) прогоняет весь цикл: собирает новые вакансии → скорит совпадение с профилем → подбирает лучший вариант резюме → генерирует tailored cover letter → отслеживает ответы HR → готовит к интервью.

Проект «живой»: он используется прямо сейчас для реального поиска работы.

## Моя роль
Solo: продуктовая идея, архитектурные решения, prompt engineering всех скиллов, тестирование на реальных вакансиях, итерации по качеству скоринга и карточек.

Весь код написан через Claude Code с детальными спецификациями — по сути, роль PM + tech lead, где «командой» является ИИ.

## Стек
- **AI-слой:** Claude Code (7 slash-commands как «скиллы» + @hr-daily агент-оркестратор + sub-agent orchestration)
- **Парсинг:** Python + Telethon (Telegram), feedparser (RSS job-boards: WeWorkRemotely, RemoteOK и др.)
- **Хранилище:** CSV-индекс (10 колонок) + per-item папки на каждую вакансию
- **Трекер контактов:** feedback_store.json — двухуровневая структура dialogs × vacancies (m2m)
- **Среда:** Windows + PowerShell, всё локально

## Что уже работает

- **/hr-collect** — запускает парсеры Telegram + RSS, скорит каждую вакансию 0–100% по профилю кандидата, принимает ≥75%, создаёт папку с полным текстом вакансии и шапкой card.md
- **/cv-match** — для каждой новой вакансии выбирает лучший вариант резюме (ru/en, product/bizdev), оценивает fit 0–100%; при fit < 70% пишет `tailoring.md` с конкретными рекомендациями по доработке CV
- **/vacancy-card** — генерирует контекст вакансии + персонализированное сообщение HR (Telegram DM, email, cover letter для ATS/формы/LinkedIn) на языке вакансии
- **/feedback-update** — запускает парсер ответов из Telegram DM, классифицирует переписку через LLM-интерпретатор, обновляет статусы (написал → жду ответа → интервью / отказ), поддерживает auto-unarchive при ответе HR на архивную вакансию
- **/interview-prep** — по вакансии со статусом «назначено интервью» генерирует: brief (контекст компании + доменная база + tricky-вопросы) и script (читать на втором экране во время звонка: «расскажи о себе», кастом-ответы, вопросы к HR)
- **/archive-vacancies** — архивирует вакансии старше 30 дней, не трогает те, у которых назначено интервью
- **@hr-daily** — оркестратор всего пайплайна: запускает 5 этапов за один вызов, этапы 2-4 идут в отдельных sub-agents чтобы не засорять контекст

## Метрики
- **137+ вакансий** отобрано агентом (≥75% совпадение с профилем) — 108 активных + 29 в архиве; всего через систему прошло 168+ с учётом удалённых вручную
- **87 откликов** отправлено через агента, **16 интервью** назначено
- **~3 часа в день** экономит пайплайн vs ручной мониторинг каналов и написание откликов

## Архитектурные решения, которыми горжусь

- **Тонкий CSV-индекс + per-item папки.** CSV хранит только 10 колонок для быстрого скана глазами; все длинные тексты (вакансия, карточка, переписка, заметки по CV) живут в `vacancies/items/{id} {роль} {компания}/`. Данные текут через диск — это позволяет sub-агентам работать независимо без передачи контекста через цепочку.

- **Slash-commands как specs, не как код.** Каждый скилл — это `.md`-файл с детальной спецификацией логики, edge cases, форматов файлов. Claude Code читает его и выполняет. Изменить поведение = отредактировать текст, не дебажить код. Быстрые итерации без страха сломать.

- **Sub-agent orchestration в @hr-daily.** Этапы 2 (сбор), 3 (CV-матч), 4 (карточки) запускаются в отдельных sub-agents: каждый агент читает свежий CSV, выполняет задачу, возвращает краткую сводку. Оркестратор не накапливает контекст от обработки десятков вакансий — работает стабильно даже при большом объёме.

- **m2m feedback_store.** Один HR-контакт (Telegram handle) может вести несколько вакансий. Реализовал двухуровневую структуру: `dialogs` (по handle) × `vacancies` (через ссылки на dialog_id). Это реальный кейс — рекрутер из большой компании отвечает сразу на несколько позиций.

- **Tiered confirmation в /feedback-update.** Рутинные переходы статуса (написал → жду ответа) применяются автоматически. Рискованные (отказ, назначено интервью, auto-unarchive) — только после подтверждения. Пайплайн не задаёт лишних вопросов, но защищает от потери данных.

## Челленджи и как решал

### 1. Качество скоринга вакансий
**Проблема:** первые версии промпта давали много false positive (вакансии с похожим названием, но нерелевантным доменом) и false negative (хорошие вакансии из смежных областей). 75% порог казался произвольным.

**Решение:** сделал скоринг многомерным — роль, домен/индустрия, формат, язык вакансии, сениорность. Каждый критерий берётся из `candidate_context.md`. Комментарий скиллa (1–2 предложения «почему подходит/нет») стал главным инструментом калибровки: читая отказы, итерировал промпт.

### 2. Засорение контекста при большом числе вакансий
**Проблема:** при обработке 10+ вакансий за прогон, каждая с полным текстом (~2-5k токенов), контекст оркестратора раздувался и качество ответов деградировало.

**Решение:** sub-agent архитектура в @hr-daily. Каждый агент-этап получает компактный промпт, работает со свежим CSV, возвращает только сводку. Оркестратор никогда не видит полных текстов вакансий.

### 3. Windows/PowerShell quirks
**Проблема:** кириллические пути ломали Glob в WSL, `$XX` в double-quoted строках PowerShell интерпретировались как переменные, Set-Content добавлял BOM в CSV и ломал Python-парсер архиватора.

**Решение:** правила зафиксированы прямо в скиллах (single-quoted строки для CSV, UTF-8 без BOM через `UTF8Encoding($false)`, fallback через PowerShell Get-ChildItem если Glob не работает). Накопленная база «ловушек» позволяет не наступать повторно.

### 4. Атрибуция сообщений при m2m диалогах
**Проблема:** когда один HR ведёт несколько вакансий, одно сообщение могло относиться к любой из них.

**Решение:** LLM-интерпретатор с prompts/interpreter.md — читает историю переписки по всем привязанным вакансиям, атрибутирует каждое новое сообщение по контексту (упоминание компании/роли), при неуверенности ставит `ambig: true` и маркирует для ручной проверки.
